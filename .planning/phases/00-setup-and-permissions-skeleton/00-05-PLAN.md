# Pitfalls this plan mitigates
- **P-PHASE0-04:** `permissions: id-token: write` at JOB level (not workflow level — keeps blast radius minimal); without this the OIDC handshake silently fails with HTTP 403.
- **P-PHASE0-05:** With Plan 01's `[tool.hatch.build.targets.wheel] packages = ["src/whatsapp_mcp"]`, `uvx whatsapp-mcp` after PyPI publish will resolve correctly. CI verifies the wheel is build-able; release.yml builds and publishes.
- **General:** Pin `astral-sh/setup-uv@v8` (not master); pin `pypa/gh-action-pypi-publish@release/v1` IF used (release.yml uses `uv publish` instead per RESEARCH.md §"Code Examples / release.yml" — both are valid).

# Trusted-publisher setup (manual, one-time, before first `git tag v0.1.0`)
This plan ships the workflow but cannot configure PyPI itself. Document in README's Development section:
1. On PyPI, navigate to `whatsapp-mcp` project (or pre-reserve via a manual upload if name is taken).
2. Project Settings → Publishing → Add a new pending publisher.
3. Owner: `<github-org>`, Repo: `whatsapp-mcp`, Workflow: `release.yml`, Environment: `pypi`.
4. Save. The first `git tag v0.1.0 && git push --tags` succeeds; subsequent tokens auto-mint.

| GitHub Actions runner → PyPI | `release.yml` publishes via OIDC; the trust boundary is the PyPI trusted-publisher config (manual, one-time, documented in README) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-15 | Information disclosure | A long-lived PyPI API token leaks via repo history, CI logs, or a contributor's local environment | mitigate | OIDC trusted-publisher: there IS no API token in the repo, the workflow, or any GitHub secret (D-17).
