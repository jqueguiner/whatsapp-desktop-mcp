Land the three MCP-decoupled guardrail modules + the shared Pydantic send-model surface. These are the structural defenses against the four account-ban / privacy / prompt-injection threat classes (T-1 fan-out, T-3 cross-chat prompt injection, T-4 audit privacy, T-5 rate-limit bypass via restart). Plan 02-03 will compose all three into `tools/send_message.py`'s body sequence per D-25.

Purpose: provide stateful + stateless guardrails that the send tool can call as opaque async functions. Persistence is load-bearing for the rate limiter (server restart MUST NOT reset the daily cap — the WhatsApp account is the protected resource, and it doesn't restart with the MCP). In-memory is correct for cross-chat-quote (per D-17 a server restart implies a fresh trust context).

Output: 4 new `.py` files (3 sender guardrails + 1 send-models module) + amended `exceptions.py` (append-only) + amended `models/__init__.py` (extend `__all__`). Zero MCP tool registrations. Zero reader.* imports. Zero filesystem coupling to WhatsApp's `ChatStorage.sqlite`.
</objective>

Helpers (verbatim per Pattern 5):
- `def _resolve_limits() -> tuple[int, int]`: reads env vars; falls back to defaults; raises `ValueError(f"WHATSAPP_DESKTOP_MCP_RATE_PER_MIN={per_min} exceeds hard max {_HARD_MAX_PER_MIN}; raising the limit risks WhatsApp account ban. - `def _ensure_db() -> Path`: creates `_DB_PATH.parent` with `parents=True, exist_ok=True`; opens connection with `f"file:{_DB_PATH}?mode=rwc"` URI; `executescript(_DDL)`; if file did not exist before this call, `os.chmod(_DB_PATH, 0o600)`; returns `_DB_PATH`.
- `def _blocking_check_and_reserve(chat_id: int, body_sha256: str) -> tuple[int, int]`: per Pattern 5 — two COUNT queries against the sliding windows; raises `RateLimitExceeded(f"Per-minute send budget exhausted: {cnt_min}/{per_min}. Retry after the oldest send in the last minute ages out.")` / `RateLimitExceeded(f"Per-day send budget exhausted: {cnt_day}/{per_day}. Retry tomorrow, or raise WHATSAPP_DESKTOP_MCP_RATE_PER_DAY (bounded {_HARD_MAX_PER_DAY}).")` when at/over budget. Returns `(per_min - cnt_min, per_day - cnt_day)` when budget remains. CRITICAL: does NOT insert — peek-and-raise only. Insertion is `record_outcome`'s job and runs AFTER the actual send attempt completes (so a cancelled send doesn't burn budget).
- `def _blocking_record(chat_id: int, body_sha256: str, outcome: str) -> None`: single INSERT into `sends` with `ts = int(time.time())`.

Public async surface (REL-02 — every DB call wraps `to_thread`):
- `async def check_and_reserve(chat_id: int, body_sha256: str) -> tuple[int, int]`: `return await asyncio.to_thread(_blocking_check_and_reserve, chat_id, body_sha256)`.
- `async def record_outcome(chat_id: int, body_sha256: str, outcome: str) -> None`: `await asyncio.to_thread(_blocking_record, chat_id, body_sha256, outcome)`.
